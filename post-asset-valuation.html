A deep-dive into building a modular, testable, and deployable natural gas contract valuation system — combining ARIMA-based price forecasting, P&L modelling, and an interactive Streamlit dashboard.

Project Summary: This project transforms a prototype Jupyter notebook into a production-ready Python package for pricing natural gas storage contracts. It combines ARIMA forecasting, a structured valuation engine, a CLI, a Streamlit dashboard, and automated pytest coverage — demonstrating a full MLOps lifecycle from research to deployment.

The Problem: Pricing a Natural Gas Storage Contract

Natural gas markets are fundamentally seasonal. Producers inject gas into underground storage facilities during low-demand summer months, then sell — or withdraw — it during peak winter demand. The arbitrage opportunity between these two market states generates profit, but only if the trade can be modelled accurately.

A natural gas storage contract defines injection dates, withdrawal dates, volumetric limits, and a set of operational costs. The core question it poses is deceptively simple: given current and forecasted prices, is this contract profitable?

Answering it requires integrating several layers of complexity: forward price curves, volume constraints, multi-month storage overhead, and variable per-unit fees. This project builds a clean, reusable tool to do exactly that.

The goal was not just to compute a number — it was to build the kind of system that a quant desk or energy trading team could actually use.

System Architecture

The project follows a layered architecture that cleanly separates research artefacts from production code — a pattern central to MLOps best practice.

The key architectural decision here is the separation of the pricing_model package from app.py. The Streamlit frontend is a consumer of the package — it does not contain any business logic. This means the same pricing engine can be invoked from the CLI, a REST API, a test suite, or the web UI with zero code duplication.

Module 1 — The ARIMA Price Forecaster

The NaturalGasForecaster class ingests the Nat_Gas.csv dataset and fits an ARIMA(1,1,1) model on historical monthly prices. ARIMA is well-suited here because natural gas prices exhibit a combination of auto-correlation (prices this month are correlated with last month's) and non-stationarity (long-run trend), which the integrated differencing term I(1) corrects for.

Why ARIMA(1,1,1)?

  • AR(1) — AutoRegressive component of order 1 captures the momentum in prices: high prices this month increase the probability of high prices next month.
  • I(1) — First-order differencing makes the series stationary, a prerequisite for ARIMA to produce valid forecasts.
  • MA(1) — Moving Average component smooths out short-run shocks in the residuals.
pricing_model/forecaster.py Python 
from statsmodels.tsa.arima.model import ARIMA
import pandas as pd

class NaturalGasForecaster:
    def __init__(self, csv_path: str):
        self.df = pd.read_csv(csv_path, parse_dates=['Dates'])
        self.df = self.df.sort_values('Dates').set_index('Dates')
        self.model = ARIMA(self.df['Prices'], order=(1, 1, 1))
        self.result = self.model.fit()

    def predict(self, target_date) -> float:
        # If within historical range, return actual price
        if target_date in self.df.index:
            return self.df.loc[target_date, 'Prices']
        # Otherwise, forecast using fitted ARIMA
        steps = len(pd.date_range(self.df.index[-1], target_date, freq='MS'))
        forecast = self.result.forecast(steps=steps)
        return forecast.iloc[-1]

    def get_historical_data(self) -> pd.DataFrame:
        return self.df.reset_index()

The predict() method elegantly handles both historical lookups and future forecasts with a single interface — callers do not need to know whether a date is in-sample or out-of-sample.

Module 2 — The Contract Valuation Engine

The ContractPricer class implements the core financial logic. It follows a standard Profit & Loss methodology for commodity storage contracts:

Net P&L Valuation Equation:
Contract Value = (Withdrawal Revenue − Withdrawal Fees) − (Injection Cost + Injection Fees) − Storage Costs − Transport Costs

Step-by-Step Valuation Process

  1. Date Normalisation — All injection and withdrawal dates are converted to pandas Timestamp objects and sorted chronologically, ensuring temporal consistency throughout the calculation.
  2. Price Discovery — For each operation date, the forecaster is queried to retrieve either the historical market price or an ARIMA-projected future price.
  3. Injection Analysis — Volume injected on each date is capped by the daily injection rate and the remaining storage capacity. Injection cost = (price × volume) + (fee × volume / 1e6).
  4. Storage Duration Overhead — Monthly storage fees accumulate over the duration between the first injection and the last withdrawal — a critical cost component often overlooked in naive models.
  5. Withdrawal Revenue — Revenue is capped by daily withdrawal limits and available inventory. Revenue = (price × volume) − (fee × volume / 1e6).
  6. Net Valuation — All revenues are summed and all costs deducted. The final number is the fair value of the contract — a positive value indicates a profitable trade.
pricing_model/contract_pricer.py Python 
def value_contract(self,
    injection_dates, withdrawal_dates,
    injection_rate, withdrawal_rate,
    max_volume, storage_cost_per_month,
    processing_fee, transport_cost) -> dict:

    total_injection_cost  = 0.0
    total_withdrawal_rev  = 0.0
    current_inventory     = 0.0

    for date in injection_dates:
        price   = self.forecaster.predict(date)
        volume  = min(injection_rate, max_volume - current_inventory)
        cost    = (price * volume) + (processing_fee * volume / 1e6)
        total_injection_cost += cost
        current_inventory    += volume

    start  = min(injection_dates)
    end    = max(withdrawal_dates)
    months = (end.year - start.year) * 12 + (end.month - start.month)
    storage_cost = storage_cost_per_month * months

    for date in withdrawal_dates:
        price    = self.forecaster.predict(date)
        volume   = min(withdrawal_rate, current_inventory)
        revenue  = (price * volume) - (processing_fee * volume / 1e6)
        total_withdrawal_rev += revenue
        current_inventory    -= volume

    contract_value = (total_withdrawal_rev - total_injection_cost
                     - storage_cost - 2 * transport_cost)

    return {
        'contract_value': contract_value,
        'total_revenue':  total_withdrawal_rev,
        'total_cost':     total_injection_cost + storage_cost + 2 * transport_cost
    }

Note the transport cost is multiplied by 2 — gas must be transported both to the storage facility and from it, representing a two-way logistics cost that is easy to omit in a naive implementation.

Module 3 — The Streamlit Dashboard

The app.py file delivers the model through an interactive Streamlit web application. The dashboard is split into two main panels: a sidebar for parameter configuration, and a main content area for scheduling and visualisation.

Key UI Features

  • Parametric sidebar — all contract constraints (max volume, injection rate, fees, transport cost) are configurable via number inputs without touching the source code.
  • Real-time P&L calculation — clicking "Calculate Contract Value" triggers the full valuation pipeline and surfaces the result with profit/loss indicators and a warning for unprofitable contracts.
  • ARIMA forecast chart — a Plotly time series chart overlays 12 months of ARIMA-projected prices on the historical data, giving traders a visual read on the forward curve before committing to a contract schedule.
  • Resource caching — the @st.cache_resource decorator ensures the ARIMA model is only fitted once per session, preventing expensive re-computation on every UI interaction.
Terminal Shell 
# Run the Streamlit dashboard locally
streamlit run app.py

# Or with a custom data source
streamlit run app.py -- --data-path /path/to/prices.csv

Testing Strategy

The tests/ directory contains a pytest suite covering the core valuation logic. Good test coverage is what allows a pricing model to be refactored, extended, or redeployed with confidence — it is the production-readiness marker that distinguishes a notebook from an engineering artefact.

tests/test_processor.py Python 
import pytest
from pricing_model.contract_pricer import ContractPricer
from unittest.mock import MagicMock

def test_profitable_contract():
    # Mock forecaster: inject at $5, withdraw at $12
    forecaster = MagicMock()
    forecaster.predict.side_effect = lambda d: 5.0 if d.month < 6 else 12.0

    pricer  = ContractPricer(forecaster)
    results = pricer.value_contract(
        injection_dates  = ['2021-01-01'],
        withdrawal_dates = ['2021-09-01'],
        injection_rate=1000000, withdrawal_rate=1000000,
        max_volume=2000000, storage_cost_per_month=100000,
        processing_fee=10000, transport_cost=50000
    )
    assert results['contract_value'] > 0

# Run the full suite
# $ pytest tests/ -v --tb=short

MLOps Lessons Learned

Design principle: A model lives in production longer than it lives in development. Structuring code as an installable package with a clean API from day one reduces the cost of every future change — CI integration, API wrapping, or model swaps become straightforward rather than risky.

This project reinforces several practical MLOps principles:

  • Separation of concerns — the forecaster, the pricer, the CLI, and the UI are independently testable and interchangeable. Swapping ARIMA for Prophet or XGBoost only requires changing forecaster.py.
  • Encapsulate complexity, expose simplicity — callers pass in dates and parameters; they do not need to know anything about ARIMA internals or pandas date arithmetic.
  • Test your business logic, not your framework — the pytest suite mocks the forecaster so that pricing logic can be verified against a synthetic price schedule, decoupling valuation correctness from forecasting accuracy.
  • Performance matters at the UI layer — @st.cache_resource ensures the ARIMA fit (which is computationally non-trivial) happens once, not on every Streamlit reruns triggered by user interaction.

Extensions and Future Work

This system is deliberately built to be extensible. Several natural next steps include:

  • Replacing ARIMA with a more sophisticated ensemble (e.g. Prophet + XGBoost residual correction) for improved long-horizon accuracy.
  • Adding a Monte Carlo simulation layer that generates a distribution of contract values under price uncertainty — moving from point estimates to probabilistic risk assessment.
  • Wrapping the pricing engine in a FastAPI REST endpoint, enabling integration with trading system dashboards or portfolio management tools.
  • Containerising the application with Docker for reproducible deployment across environments.
