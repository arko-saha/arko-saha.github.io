Containerising Apache Airflow with Astro: A Production-Ready Weather ETL

Most Airflow tutorials show you a DAG running on your laptop. I wanted something actually deployable. This post documents a real-time ingestion pipeline — Open-Meteo → PostgreSQL — containerised with Astro CLI and monitored via dBeaver.

Beyond the "Hello World" DAG

Every Airflow tutorial has the same structure: define a DAG, add two BashOperators that echo "hello" and "world", run it locally, declare victory. None of them show you how to handle a real data source, a real database connection, or a local environment that actually mirrors what you'd deploy to a cloud platform.

This project was my attempt to build the real thing. The pipeline fetches live weather data for London from the Open-Meteo API every time it runs, transforms the raw JSON into structured records, and loads them into a dedicated PostgreSQL instance — all orchestrated by Airflow, running entirely in Docker via Astronomer's Astro CLI.

Technical Stack

The stack was chosen deliberately to demonstrate a production-ready MLOps foundational layer. Before you can train models, you need reliable data ingestion — and that means treating your pipelines like software, not ad-hoc scripts.

Stack: Apache Airflow (TaskFlow API) · Astronomer Astro CLI · Docker & Docker Compose · PostgreSQL 12 · Open-Meteo API (free, no key) · dBeaver (monitoring)

The choice of Astro CLI over a vanilla Airflow Docker Compose setup is significant. Astro manages the container lifecycle, handles the scheduler/webserver/triggerer topology automatically, and provides a local dev experience that is much closer to what Astronomer's managed cloud deployment looks like. Fewer surprises when you go to production.

Pipeline Architecture

The pipeline follows a classic ETL pattern with three distinct task functions, connected using Airflow's TaskFlow API (the @task decorator model introduced in Airflow 2.0):

  1. Extract — A Python task calls the Open-Meteo API at London's coordinates (51.5°N, -0.1°W) and returns the raw JSON response. The task uses response.raise_for_status() to fail hard on HTTP errors — intentionally. Airflow's retry mechanism handles transient failures; partial data entering the transform step is worse than no data at all.
  2. Transform — The JSON is parsed to extract five key metrics: temperature (°C), windspeed (km/h), wind direction (degrees), WMO weather code, and a server-side timestamp. The result is a typed Python dict — no ORM, no dataframe overhead. Keep the transform layer thin and explicit.
  3. Load — The structured record is inserted into a weather_data table in the postgres-db container using Airflow's PostgresHook. The hook resolves the connection from Airflow's encrypted connections store — credentials never appear in the DAG file.

The DAG Code

The TaskFlow API (@task decorator model, introduced in Airflow 2.0) eliminates the boilerplate of the older PythonOperator pattern and makes XCom data passing implicit — tasks return values and receive them as typed arguments, exactly as you'd write normal Python functions.

dags/weather_etl.py Python 
from airflow.decorators import dag, task
from airflow.providers.postgres.hooks.postgres import PostgresHook
from datetime import datetime
import requests

@dag(
    dag_id="weather_etl_pipeline",
    schedule_interval="@hourly",
    start_date=datetime(2024, 1, 1),
    catchup=False,          # critical: prevents backfill flood on first run
    tags=["etl", "weather"],
)
def weather_etl():

    @task()
    def extract() -> dict:
        url = (
            "https://api.open-meteo.com/v1/forecast"
            "?latitude=51.5&longitude=-0.1"
            "&current_weather=true"
        )
        response = requests.get(url, timeout=10)
        response.raise_for_status()   # fail hard; let Airflow handle retry
        return response.json()

    @task()
    def transform(raw: dict) -> dict:
        cw = raw["current_weather"]
        return {
            "timestamp":       cw["time"],
            "temperature_c":   cw["temperature"],
            "windspeed_kmh":   cw["windspeed"],
            "wind_direction":  cw["winddirection"],
            "weather_code":    cw["weathercode"],
        }

    @task()
    def load(record: dict) -> None:
        hook = PostgresHook(postgres_conn_id="postgres_default")
        hook.run(
            """
            INSERT INTO weather_data
              (timestamp, temperature_c, windspeed_kmh,
               wind_direction, weather_code)
            VALUES
              (%(timestamp)s, %(temperature_c)s, %(windspeed_kmh)s,
               %(wind_direction)s, %(weather_code)s)
            ON CONFLICT (timestamp) DO NOTHING
            """,
            parameters=record
        )

    # XCom passing is implicit — return values flow as typed arguments
    raw  = extract()
    rec  = transform(raw)
    load(rec)

weather_etl()

Docker & Container Setup

The Astro CLI manages four services when you run astro dev start. Understanding each container's role is essential when debugging: a failed DAG run could be a scheduler issue, a database connectivity problem, or a Python dependency conflict — and the logs live in different containers.

ContainerRolePortWhen to check
webserverAirflow UI8080DAG visibility, task logs
schedulerDAG orchestration & trigger—DAG not running? Start here
postgres-dbTarget database (custom name)5432Data not persisting
triggererDeferrable operators—Async sensor tasks

The PostgreSQL container is explicitly named postgres-db rather than the Astro default of postgres. This matters in development: when multiple Docker Compose projects are running, the default name collides. A custom name eliminates ambiguity and makes the docker exec commands unambiguous across projects.

# Extends the official Astronomer Airflow image
FROM quay.io/astronomer/astro-runtime:8.8.0

# System packages for DAG dependencies (e.g. psycopg2-binary native build)
RUN apt-get update && apt-get install -y --no-install-recommends curl libpq-dev

Production Pattern: Idempotent Loads

One gap in most ETL tutorials is idempotency. A simple INSERT is not safe to retry: if the DAG fails mid-run and retries, duplicate rows will accumulate silently. Production load tasks must be idempotent — running them twice produces the same result as running them once.

For time-series data, the correct pattern is a unique constraint on the natural key (timestamp in this case) combined with an ON CONFLICT DO NOTHING clause. This is shown in the DAG code above. It makes the load task safe to retry at any point in the execution window — a property Airflow's retry mechanism implicitly depends on.

Live Monitoring with dBeaver

Once data is in Postgres, you need a way to inspect it that doesn't require CLI access. I connected dBeaver — a free, cross-platform SQL client — directly to the containerised database. For demos and stakeholder reviews, seeing rows appear in a GUI is far more convincing than tailing Docker logs.

dBeaver connection settings:
Host: localhost · Port: 5432 · Database: postgres
Username: postgres · Password: postgres

The monitoring query is trivial — but seeing it update on each hourly run makes the pipeline feel alive:

SELECT timestamp, temperature_c, windspeed_kmh, weather_code
FROM weather_data
ORDER BY timestamp DESC
LIMIT 24;  -- last 24 hours

Three Engineering Lessons

Building this against actual production patterns surfaced three things that tutorial setups consistently obscure:

1. PostgresHook over raw psycopg2. Using Airflow's PostgresHook keeps credentials out of your DAG code entirely. The connection lives in Airflow's encrypted connections store, configured once in the UI or via an environment variable. Credential rotation never requires a DAG change — the operational discipline this enables is significant at scale, and it's the correct pattern for any secrets management in Airflow.

2. TaskFlow API makes data flow explicit. The @task decorator model makes pipeline data flow visible through Python function signatures. Compare this to the old PythonOperator(python_callable=fn) approach, where XCom pushes and pulls happen via side-effect calls inside the task body — invisible, fragile, and hard to test. The TaskFlow model is composable, testable, and reads exactly like normal Python. It's the right pattern for any pipeline you expect to maintain long-term.

3. catchup=False is not optional. Forget this on your first run and Airflow will attempt to backfill every scheduled interval from start_date to now — potentially hundreds of API calls in rapid succession. More critically, if your pipeline does anything stateful (database writes, notifications, downstream triggers), that flood will corrupt state in ways that are difficult to diagnose. Set catchup=False as a default on every DAG unless you have an explicit, idempotent backfill requirement.

The discipline of treating ETL pipelines as software — with containers, connection stores, idempotent loads, and proper observability — pays off the first time you have to debug a production incident at 2am.
